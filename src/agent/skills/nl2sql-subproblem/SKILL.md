# NL2SQL 子问题分解智能体

## 概述

子问题分解智能体是 SQL-of-Thought 流水线的**第2步**。给定自然语言问题和 Schema 关联输出，它将查询分解为**子句级子问题**，以结构化 JSON 键值对的形式表示。

这种分解提供了查询意图的模块化表示，使下游智能体（查询计划、SQL 生成）能够在较小的、定义明确的单元上进行推理，而非一次性处理整个查询。

## 在流水线中的角色

```
输入：  自然语言问题 (Q) + 裁剪后的 Schema (S)
输出：  包含子句级子问题的结构化 JSON 对象 (C)
下一步：→ nl2sql-query-plan（第3步）
```

## 支持的子句类型

每个识别出的子句以键值对形式表示，其中**键**为 SQL 子句类型，**值**为该子句应实现的目标的自然语言描述：

| 子句键 | 描述 | 示例值 |
|--------|------|--------|
| `WHERE` | 行级过滤条件 | "薪资大于 50000 的员工" |
| `GROUP BY` | 分组/聚合维度 | "按部门名称分组" |
| `HAVING` | 聚合后过滤 | "员工人数超过 10 人的组" |
| `JOIN` | 表关联需求 | "通过 dept_id 连接员工表和部门表" |
| `DISTINCT` | 去重需求 | "唯一的部门名称" |
| `ORDER BY` | 排序规则 | "按平均薪资降序排列" |
| `LIMIT` | 结果数量限制 | "薪资最高的前5名" |
| `EXCEPT` | 集合差操作 | "部门中排除没有员工的那些" |
| `UNION` | 集合并操作 | "合并美国和欧盟的员工数量" |

> **注意**：并非每个查询都会出现所有子句。仅包含基于问题分析实际需要的子句。

## 执行流程

### 第1步：分析问题意图

阅读自然语言问题并识别其中隐含的 SQL 子句：

```
问题："查询人数超过10人的部门中员工的平均薪资"

意图分析：
- "查询平均薪资" → 需要带聚合函数的 SELECT（AVG），需要 GROUP BY
- "部门中的员工" → 需要通过外键连接 employees 和 departments 表
- "人数超过10人" → 需要 HAVING 子句（聚合后过滤）
- 未明确提及 ORDER BY、LIMIT、DISTINCT、EXCEPT 或 UNION
```

### 第2步：将意图映射为子句级子问题

对每个识别出的子句，编写该子句应实现的目标的**自然语言描述**。请勿在值中编写 SQL 语法——使用描述性语言：

```
子问题：
{
  "SELECT": "每个符合条件的部门的平均薪资",
  "JOIN": "通过 dept_id 连接 employees 表和 departments 表",
  "GROUP BY": "按部门分组",
  "HAVING": "仅包含员工人数大于10的部门"
}
```

### 第3步：对照 Schema 验证

将每个子问题与裁剪后的 Schema 交叉检查，确保可行性：
- 提及的列能否在关联表中找到？
- 连接路径是否与已识别的外键一致？
- 聚合类型是否受可用列支持？

### 第4步：格式化输出

## 输出模板

```json
{
  "SELECT": "要检索内容的描述",
  "WHERE": "行级过滤条件的描述（如有需要）",
  "JOIN": "需要连接哪些表以及如何连接的描述（如有需要）",
  "GROUP BY": "分组维度的描述（如有需要）",
  "HAVING": "聚合后过滤条件的描述（如有需要）",
  "DISTINCT": "去重需求的描述（如有需要）",
  "ORDER BY": "排序需求的描述（如有需要）",
  "LIMIT": "结果数量限制的描述（如有需要）",
  "EXCEPT": "集合差操作的描述（如有需要）",
  "UNION": "集合并操作的描述（如有需要）",
  "reasoning": "逐步分解逻辑……"
}
```

## 示例

### 简单查询
```
问题："所有员工的姓名是什么？"
→ {
  "SELECT": "来自 employees 表的员工姓名"
}
```

### 复杂聚合查询
```
问题："查询拥有至少5名员工的部门中平均薪资最高的部门，按平均薪资降序排列，只显示前3名"

→ {
  "SELECT": "部门名称和平均薪资",
  "JOIN": "通过 dept_id 连接 employees 和 departments 表",
  "GROUP BY": "按部门分组",
  "HAVING": "拥有至少5名员工的部门",
  "ORDER BY": "按平均薪资降序排列",
  "LIMIT": "前3条结果"
}
```

### 集合操作查询
```
问题："显示所有属于 Electronics 类别但不在已停产产品列表中的产品"

→ {
  "SELECT": "产品名称和详细信息",
  "WHERE": "属于 Electronics 类别的产品",
  "EXCEPT": "排除出现在已停产产品列表中的产品"
}
```

## 最佳实践

- **值中使用自然语言，而非 SQL 语法**：值应当描述意图，而非具体实现
- **仅包含相关子句**：如果问题未隐含某个子句，则完全省略它
- **描述要具体**："员工人数超过10人的部门"优于"对部门进行某种过滤"
- **交叉引用 Schema**：确保子问题中提到的每个列/表都在关联的 Schema 中存在
- **考虑隐含子句**：有些查询即使未明确说明也隐含了 GROUP BY（例如"每个部门的平均值" → GROUP BY）

## 常见陷阱

| 陷阱 | 纠正方式 |
|------|---------|
| 在子问题的值中编写 SQL 语法 | 仅使用自然语言描述 |
| 遗漏隐含的 GROUP BY | "每X的平均值"或"每Y的数量"始终隐含 GROUP BY |
| 混淆 WHERE 和 HAVING | WHERE = 行级过滤；HAVING = 分组级过滤 |
| 引用多个表时遗漏 JOIN | 任何跨表引用列的查询都需要 JOIN |
| 添加不必要的子句 | 仅包含显式或隐式要求的子句 |
