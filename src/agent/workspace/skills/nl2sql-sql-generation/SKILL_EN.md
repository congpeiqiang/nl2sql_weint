# NL2SQL SQL Generation Agent

## Overview

The SQL Generation Agent is **Step 4** in the SQL-of-Thought pipeline. It consumes the natural language question and the **procedural query plan** to synthesize an executable SQL query.

This is the **only agent in the entire pipeline** that produces SQL. Post-processing removes extraneous artifacts to ensure syntactic validity.

## Role in Pipeline

```
Input:  Natural language question (Q) + Procedural query plan (P) + Cropped schema (S)
Output: Executable SQL query (post-processed)
Next:   → Execute SQL against database → SUCCESS (end) or ERROR (→ nl2sql-correction)
```

## Process

### Step 1: Review the Query Plan

Carefully read the procedural plan generated by the Query Plan Agent. Understand each numbered step and the reasoning behind it. The plan is the primary guide for SQL synthesis — NOT the raw question alone.

### Step 2: Map Plan Steps to SQL Constructs

Translate each procedural step into its corresponding SQL construct:

| Plan Pattern | SQL Construct |
|-------------|---------------|
| "Start from X table" | `FROM X` |
| "Join with Y on condition" | `JOIN Y ON condition` |
| "Filter rows where..." | `WHERE condition` |
| "Group by..." | `GROUP BY column(s)` |
| "Filter groups where..." | `HAVING condition` |
| "Compute AVG/COUNT/SUM..." | `AVG()/COUNT()/SUM() in SELECT` |
| "Order by... descending" | `ORDER BY col DESC` |
| "Limit to top N" | `LIMIT N` |
| "Only distinct..." | `DISTINCT` |
| "Except/remove..." | `NOT IN (subquery)` or `EXCEPT` |

### Step 3: Assemble SQL Following Plan Order

Build the SQL query step by step, following the plan's numbered sequence:

```
Plan:
1. Join employees with departments on e.dept_id = d.id
2. Group by d.dept_name
3. Compute AVG(e.salary) and COUNT(*)
4. Filter groups where COUNT(*) > 10
5. Return dept_name and avg_salary

→ SQL Construction:
SELECT d.dept_name, AVG(e.salary) AS avg_salary   -- Step 5: SELECT columns
FROM employees e                                     -- Step 1a: FROM
JOIN departments d ON e.dept_id = d.id               -- Step 1b: JOIN
GROUP BY d.dept_name                                 -- Step 2: GROUP BY
HAVING COUNT(*) > 10                                 -- Step 4: HAVING
```

### Step 4: Post-Processing

Apply the following post-processing rules to ensure clean, valid SQL:

| Artifact | Action |
|----------|--------|
| Trailing semicolons (`;`) | **Remove** — SQL should not end with semicolons for execution |
| Natural language fragments in output | **Remove** — Strip any prose that leaked into the SQL |
| Inline comments (`-- ...`) | **Remove** — Clean SQL without comments for execution |
| Multiple whitespace/newlines | **Normalize** — Collapse to single spaces for readability |
| Lowercase table/column names | **Keep as-is** — Respect the schema's naming convention |

### Step 5: Validate Against Schema

Quick structural check:
- All referenced tables exist in the cropped schema
- All referenced columns exist in their respective tables
- JOIN conditions use valid FK relationships
- Aggregation columns appear in GROUP BY or are aggregates themselves

## Generation Rules

1. **Temperature = 0**: Always use deterministic generation for SQL
2. **Follow the plan faithfully**: Do not improvise beyond what the plan specifies
3. **Use SQLite-compatible syntax**: The target is SQLite (as per Spider benchmark), so:
   - No `LIMIT/OFFSET` syntax variations beyond `LIMIT n`
   - Use `EXCEPT` instead of `MINUS`
   - String comparisons are case-sensitive by default
4. **Alias consistently**: Use short, consistent table aliases (e.g., `e` for `employees`, `d` for `departments`)
5. **Explicit column references**: Always qualify columns with table alias when multiple tables are involved (`e.salary`, not just `salary`)
6. **No clause-specific hard-coded rules**: Do NOT add special prompt rules for individual clauses (JOIN, LIMIT, etc.) — this inflates context and distracts the model

## Common SQL Patterns

### Simple Selection
```sql
SELECT T1.name FROM employees AS T1
```

### Join + Aggregation + HAVING
```sql
SELECT T1.dept_name, AVG(T2.salary) AS avg_salary
FROM departments AS T1
JOIN employees AS T2 ON T1.dept_id = T2.dept_id
GROUP BY T1.dept_name
HAVING COUNT(*) > 10
```

### Nested Subquery
```sql
SELECT T1.name, T1.salary
FROM employees AS T1
WHERE T1.salary > (
    SELECT AVG(T2.salary)
    FROM employees AS T2
    WHERE T2.dept_id = T1.dept_id
)
```

### Set Operations
```sql
SELECT T1.product_name FROM products AS T1
WHERE T1.category = 'Electronics'
EXCEPT
SELECT T2.product_name FROM discontinued_products AS T2
```

## Common Pitfalls

| Pitfall | Correction |
|---------|-----------|
| Generating SQL without following the plan | The plan is the primary guide — always map plan steps to SQL |
| Adding trailing semicolons | Remove in post-processing |
| Unqualified column names in multi-table queries | Always use table aliases (T1.col, T2.col) |
| Missing GROUP BY when using aggregates | Every non-aggregated column in SELECT must appear in GROUP BY |
| Confusing HAVING vs WHERE | WHERE = row filter (before grouping); HAVING = group filter (after grouping) |
| Using non-SQLite syntax | Stick to SQLite-compatible syntax |
| Adding clause-specific hard-coded rules | This was proven to reduce accuracy — keep prompts generic |
